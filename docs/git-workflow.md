# Git — рабочий процесс и соглашения

**Git** — система контроля версий, которая позволяет отслеживать изменения в коде, работать в команде и управлять разными версиями проекта.
Соблюдение единого подхода к работе с Git делает взаимодействие в команде предсказуемым и снижает риск конфликтов.

## 1. Установка Git

**Linux (Ubuntu/Debian)**
```bash
sudo apt update
sudo apt install git
```

**macOS**
```bash
brew install git
```

**Windows**  
Скачайте установщик с [официального сайта](https://git-scm.com/).

**После установки проверьте:**
```bash
git --version
```


## 2. Настройка Git
Перед началом работы укажите имя и email — они будут привязаны к вашим коммитам.

**Хорошо** [🟢]
```bash
git config --global user.name "Ivan Petrov"
git config --global user.email "ivan.petrov@example.com"
```

**Плохо** [🔴]
```bash
# без настройки или с выдуманными данными
git config --global user.name "a"
```

**Проверить настройки:**
```bash
git config --list
```


## 3. Инициализация проекта (git init)  
Если вы начинаете новый проект с нуля и хотите отслеживать его через Git, первым делом нужно создать локальный репозиторий.

**Пошагово:**
1) Создайте папку проекта (если её ещё нет):
    ```bash
    mkdir my-project
    cd my-project
    ```
2) Инициализируйте Git-репозиторий:
    ```bash
    git init
    ```
3) Git создаст скрытую папку `.git` — там будет храниться вся история изменений.

4) После инициализации можно добавлять файлы и делать первый коммит:
    ```bash
    echo "# My Project" > README.md
    git add README.md
    git commit -m "Initial commit"
    ```
5) (Опционально) Свяжите локальный репозиторий с удалённым (например, на `GitHub`):
    ```bash
    git remote add origin https://github.com/username/my-project.git
    git push -u origin main
    ```

**Хорошо** [🟢]  
```bash
# Создали новый проект
mkdir awesome-bot
cd awesome-bot
git init
git add .
git commit -m "Initial commit"
```

**Плохо** [🔴]
```bash
# Работать без git init, просто хранить файлы в папке
# Нет истории, нет возможности отката, нельзя работать в команде
```

**Когда НЕ нужно использовать `git init`:**  
- Если вы клонируете существующий репозиторий через `git clone` — там `.git` уже есть.
- Если вы работаете через `fork` — вы клонируете, а не создаёте с нуля.


## 4. Файл .gitignore  
Файл `.gitignore` указывает `Git`, какие файлы и папки не нужно отслеживать и включать в репозиторий. Это обязательный файл для любого проекта.

**Что обычно игнорируют:**  
- Временные файлы (`*.tmp`, `*.log`, `*.cache`)
- Зависимости (`node_modules/`, `venv/`, `__pycache__/`)
- Файлы с секретами (`.env`, `config.local.*`)
- Системные файлы (`.DS_Store`, `Thumbs.db`)
- Результаты сборки (`dist/`, `build/`, `*.pyc`)

**Пример для Python-проекта:**

**Хорошо** [🟢]  
```gitignore
# Виртуальное окружение
venv/
env/
ENV/

# Кэш Python
__pycache__/
*.pyc
*.pyo
*.pyd

# Файлы IDE
.vscode/
.idea/
*.swp

# Секреты и конфиги
.env
.secrets
config.local.py

# Логи
*.log
*.tmp

# Системные файлы
.DS_Store
Thumbs.db

# Файлы сборки
dist/
build/
*.egg-info/
```

**Плохо** [🔴]  
```gitignore
# Игнорировать вообще всё
*
# Или не использовать .gitignore совсем
```

**Как создать и использовать:**
1) Создать файл в корне проекта:
    ```bash
    touch .gitignore
    ```
2) Добавить правила (как в примере выше)

3) Проверить, что файлы игнорируются:
    ```bash
    git status
    # Файлы из .gitignore НЕ должны показываться как untracked
    ```
4) Закоммитить `.gitignore`:
    ```bash
    git add .gitignore
    git commit -m "Add .gitignore for Python project"
    ```

**Готовые шаблоны:**  
Вместо того чтобы писать всё вручную, можно использовать готовые шаблоны с [gitignore.io](https://www.toptal.com/developers/gitignore) или в репозитории [github/gitignore](https://github.com/github/gitignore):  
```bash
# Пример: скачать шаблон для Python
curl -o .gitignore https://raw.githubusercontent.com/github/gitignore/main/Python.gitignore
```

**Важные нюансы:**
- `.gitignore` сам должен быть добавлен в `Git` (его нужно коммитить)
- Если файл уже отслеживается `Git`, добавление его в `.gitignore` не сработает — нужно сначала удалить из индекса:
    ```bash
    git rm --cached secret.env
    ```
- Можно создавать `.gitignore` в подпапках — правила будут действовать только для этой папки и её содержимого.


## 5. Основные понятия

- `Репозиторий` — хранилище кода и истории изменений.
- `Локальный репозиторий` — копия проекта на вашем компьютере.
- `Удалённый репозиторий` — репозиторий на сервере (например, `GitHub`, `GitLab`).
- `Ветка` (`branch`) — отдельная линия разработки.
- `Коммит` (`commit`) — сохранение изменений с комментарием.
- `Fork` — копия чужого репозитория на ваш аккаунт.
- `Pull request` (`PR`) — запрос на вливание ваших изменений в исходный репозиторий.


## 6. Клонирование репозитория

**Скопировать удалённый репозиторий к себе на компьютер:**
```bash
git clone https://github.com/username/repo.git
```

**После клонирования перейдите в папку проекта:**
```bash
cd repo
```

**Хорошо** [🟢]  
Клонировать через `SSH` (если настроен ключ):  
```bash
git clone git@github.com:username/repo.git
```

**Плохо** [🔴]  
Клонировать вручную через скачивание `ZIP` — теряется история и связь с удалённым репозиторием.


## 7. Работа через fork  
Используется, когда у вас нет прав на запись в чужой репозиторий, но вы хотите предложить изменения.

**Пошагово:**
1) На странице репозитория нажмите `Fork` (в правом верхнем углу `GitHub`).
2) Склонируйте свой форк:
    ```bash
    git clone https://github.com/ваш_логин/repo.git
    ```
3) Добавьте ссылку на оригинальный репозиторий (`upstream`):
    ```bash
    cd repo
    git remote add upstream https://github.com/оригинал/repo.git
    ```
4) Проверьте удалённые репозитории:
    ```bash
    git remote -v
    ```
    Должно быть:
    ```text
    origin    https://github.com/ваш_логин/repo.git (fetch)
    origin    https://github.com/ваш_логин/repo.git (push)
    upstream  https://github.com/оригинал/repo.git (fetch)
    upstream  https://github.com/оригинал/repo.git (push)
    ```


## 8. Ветки  
Все изменения делайте в отдельных ветках, не в `main/master`.

**Создать ветку и переключиться:**
```bash
git checkout -b feature/add-login
```

**Посмотреть список веток:**
```bash
git branch
```

**Переключиться на другую ветку:**
```bash
git checkout main
```

**Хорошо** [🟢]  
Ветка с осмысленным именем:  
```bash
git checkout -b bugfix/fix-null-pointer
```

**Плохо** [🔴]
```bash
git checkout -b new
```


## 9. Обновление проекта (pull)  
Перед началом работы и перед созданием `pull request` обязательно обновите свою ветку относительно оригинального репозитория.

**Если вы работаете через fork:**  
1) Переключитесь в `main`:  
    ```bash
    git checkout main
    ```
2) Заберите изменения из `upstream`:  
    ```bash
    git pull upstream main
    ```
3) Обновите свой форк на GitHub:
    ```bash
    git push origin main
    ```
4) Переключитесь в свою рабочую ветку и влейте туда обновлённый `main`:
    ```bash
    git checkout feature/add-login
    git merge main
    ```

**Хорошо** [🟢]  
Регулярно делать pull, чтобы избежать конфликтов.

**Плохо** [🔴]  
Работать в старой ветке неделями без синхронизации.


## 10. Коммиты  
Коммитите часто, но логически завершёнными кусками.

**Посмотреть изменения:**
```bash
git status
git diff
```

**Добавить файлы в коммит:**
```bash
git add file.py          # конкретный файл
git add .                # все изменения (осторожно)
```

**Сделать коммит:**
```bash
git commit -m "Краткое описание изменений"
```

**Хорошо** [🟢]  
Понятное сообщение в повелительном наклонении:
```bash
git commit -m "Add user authentication"
git commit -m "Fix crash when loading empty config"
```

**Плохо** [🔴]  
```bash
git commit -m "fix"
git commit -m "123"
```


## 11. Отправка изменений (push)  
**После коммитов отправьте вашу ветку в `ваш форк`:**
```bash
git push origin feature/add-login
```
Если ветки нет на удалённом репозитории, `Git` предложит создать её.


## 12. Pull request

**Когда всё готово:**  
1) Зайдите в ваш форк на `GitHub`.

2) `GitHub` обычно сам предлагает создать `Pull Request` после свежего пуша. Если нет — нажмите `Pull requests` → `New pull request`.

3) Выберите:
- base repository: оригинальный репозиторий, ветка `main`
- head repository: ваш форк, ваша ветка

4) Напишите понятное описание изменений.

5) Нажмите `Create pull request`.

**После создания PR:**  
- Дождитесь проверки (code review).
- Если нужны правки — сделайте новые коммиты в той же ветке и запушите: `PR` обновится автоматически.
- После одобрения ваш `PR` вольют (или вы нажмёте `Merge pull request`).

**Хорошо** [🟢]  
В описании `PR` указать, что сделано и зачем.
Привязать issue: `Closes #42`.

**Плохо** [🔴]
Пустой PR без описания.


## 13. Правила именования веток и коммитов

**Ветки:** 
- `feature/название` — новая функциональность
- `bugfix/название` — исправление ошибки
- `hotfix/название` — срочное исправление на прод
- `refactor/название` — рефакторинг

**Хорошо** [🟢]  
```text
feature/add-password-reset
bugfix/fix-login-crash
hotfix/security-patch-2025
```

**Плохо** [🔴]  
```text
new
fix1
test
```

**Коммиты:**  
Правило: повелительное наклонение, не длиннее 50 символов (для заголовка).

**Хорошо** [🟢] 
```text
Add validation for email field
Remove deprecated API call
Fix typo in README
```

**Плохо** [🔴]  
```text
fixed bug
some changes
123
```
